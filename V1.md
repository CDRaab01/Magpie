# V1.md — the tiered path to v1 (supersedes CLAUDE.md §10's phase list for remaining work)

> Written 2026-07-05 after the post-Phase-8 review; **amended same day after a deep code
> review of the correctness core** (`app/ledger/`, `app/rules/`, `app/imports/`,
> `app/ingest/`, the wiring services, and dragonfly-id's smoke endpoint) — see "Code-review
> findings" below; each finding is folded into a tier. The phase list got the skeleton built
> and deployed; what remains is dependency-ordered, not sequential, so it's tiers from here.
> **A tier is a dependency bucket**: items within a tier can run in parallel / any order, but
> starting tier N+1 before tier N is done either wastes work or (Tier 0) violates a gate.
> `[H]` = human-gated (needs the owner's hands, credentials, or eyes — Claude can assist but
> not complete). Everything here inherits CLAUDE.md's locked decisions (§2) and guardrails
> (§6, §8) unchanged.
>
> **v1 is done when:** statement parity holds for two consecutive months (§9 gate), the owner
> runs the review queue daily from the phone without touching an API client, a simulated
> missing bill and a short paycheck both page the phone, and every screen passes the Tier 4
> UI/UX bar.

## Bootstrap — how the next session starts this work

1. **Read, in order:** host `C:\Code\CLAUDE.md` (suite invariants — signing key, epoch-minute
   versionCodes, compose `environment:` rule, no `pull_request` on self-hosted runners) →
   this repo's `CLAUDE.md` §2 (locked decisions — do not relitigate) → this file → the
   status header of `ARCHITECTURE.md` (as-built + known gaps).
2. **Verify live state before changing anything:** `docker ps` shows `magpie-server-1` /
   `magpie-db-1` up; `GET 127.0.0.1:8005/health` + `/version` (commit should match `main`);
   latest GitHub Actions runs green (check jobs, not the rollup); `git log origin/main..main`
   — **if local commits are unpushed, reconcile that with the owner first**, don't build on
   an undecided base.
3. **Respect the gates:** Tier 0 items are owner-authorized only. Specifically: no real
   financial data (backfill, IMAP creds) before the NAS-dump encryption lands; no dragonfly-id
   or Pulse or host-script change without a separate explicit go-ahead; new secrets in a live
   `.env` need their own confirmation even under a broad "go ahead".
4. **Start with Tier 1 in this order:** F1 (balance anchoring — pure math, well-specified,
   the parity gate depends on it) → F3+F12 (transfer guards + un-pair) → F6/F7/F8/F16
   (rule/parser fixes) → seeded categories (migration) → review-queue correct flow + Settings
   category editor (Android). **Each F-fix begins with a regression test that reproduces the
   bug** — the finding's "failure scenario" column is the test case.
5. **Work discipline (proven across Phases 0–8):** server tests against a throwaway DB
   (`127.0.0.1`, never localhost; `DB_NULLPOOL`); `ruff check` AND `ruff format --check`
   before pushing; Android `testDebugUnitTest` + `assembleDebug` locally; ARCHITECTURE.md
   updated **in the same commit** as the code; after push, verify CI job-by-job, then confirm
   the auto-deploy via `/version`. Public repo: sentinel-only fixtures, no personal PII —
   when a scanner flags one file, grep the rest of the commit for the same pattern.
6. **Multi-agent note:** if parallelizing, each agent works in its own worktree off `main`
   (sibling directory + copy `local.properties`), never the shared checkout.

| # | Sev | Finding | Fix in |
|---|-----|---------|--------|
| F1 | **Critical** | **Balance/checkpoint math can't reconcile.** `account_service.compute_account_balance` sums *every* transaction ever and compares against the latest checkpoint's stated balance. Unless the ledger holds the account's complete lifetime history from $0, the delta is never zero — after a 12-month backfill every account reads "Off by $X" (X = the balance 12 months ago) forever, and the statement-parity acceptance gate is mathematically unreachable. Spec §4 says balance is computed *between* checkpoints: derived = earliest checkpoint's stated balance + signed sum of transactions dated after it (and the delta compares that at the latest checkpoint). | Tier 1 |
| F2 | **Critical (hardening)** | **Pin the smoke-token subject email.** dragonfly-id's `POST /smoke/token` accepts a caller-supplied `subject_email`; §9's original design called for "a designated throwaway smoke email". Add an allowlist (e.g. `SMOKE_SUBJECT_EMAILS`) in dragonfly-id config and reject others, per least-privilege. Shared prod identity server — needs its own go-ahead. | Tier 0 |
| F3 | **High** | **Transfer matching pairs any ±equal amounts.** `_find_transfer_partner` matches on exact opposite amount + different account + 3-day window with **no kind/status/review-state/merchant guard** — a $50 card spend and a $50 Zelle deposit into checking become a "transfer", both silently rewritten to `review_state="auto"` and excluded from all rollups; the partner is mutated **even if it was human-confirmed**. Round-amount collisions are common in real data. Guards needed: payment-shaped signals (merchant keywords / kind expectations), never silently flip a `confirmed` row (route to review instead), and an un-pair action (F12). | Tier 1 |
| F4 | **High** | **Rollups will double-count the moment email + CSV both run.** Email rows stay `status="pending"` forever, CSV creates independent `posted` rows, and `rollup_month`/`list_transactions` have no status logic — every emailed swipe counts twice in the Home panel after each reconciliation. Same root as pending→posted matching (Tier 1 #7); named separately because it corrupts the headline numbers, not just the row list. | Tier 1 |
| F5 | **High** | **Card CSV sign inversion is near-certain at backfill.** Amex (notoriously) exports charges as *positive* amounts; the generic parser + `default_kind` from sign will import an entire card history inverted (charges as income). The spec's "per-institution column mappings" were never built — add a per-import (or per-account-institution) mapping incl. a sign-flip flag, validated against the real files. | Tier 2 |
| F6 | **Medium** | **Recurring rules permanently derail after one miss.** `last_matched_at` advances only on a successful auto-file; a single out-of-band/missed occurrence routes to review, the human confirms, but the rule's window never advances — every subsequent occurrence is "outside expected cadence window" forever. Also `last_matched_at` is set to wall-clock `now`, not the matched transaction's date, skewing windows for delayed imports. Fix: confirming a rule-flagged transaction advances its rule; store the txn date. | Tier 1 |
| F7 | **Medium** | **Zelle direction unguarded.** `parse_usbank` treats any "Zelle payment" subject as `income` with a positive amount — a "You sent a Zelle payment" alert would book outbound money as income. Require the received/deposited template explicitly; parse "sent" as spend or raise `UnparsedEmail`. | Tier 1 |
| F8 | **Medium** | **Merchant normalization strips bare prefixes.** `^(SQ\|TST\|PAYPAL\|SP\|POS\|AMZN\|WL)\s*\*?\s*` has all-optional separators, so it fires mid-word: SPOTIFY→"OTIFY", POSTAL SERVICE→"TAL SERVICE", POSTMATES→"TMATES". Combined with two-way substring containment (observed "AMAZON" matches rule "AMAZON PRIME"), rules will mis-fire across unrelated merchants. Require a real separator after the prefix; consider one-way containment. | Tier 1 |
| F9 | **Medium** | **CSV dedupe collapses legitimate duplicates.** The (account, date, amount, description) fingerprint skips the *second* identical same-day transaction (two $5.00 coffees) — real rows vanish and parity fails confusingly. Make dedupe multiplicity-aware (count occurrences in file vs. DB). | Tier 2 |
| F10 | **Medium** | **Budgets are global, not user-scoped.** `Budget` has no `user_id` and `list_budgets` doesn't filter — deliberate "single-user" shortcut, except the smoke client's throwaway user **is a second user in prod**, able to read/write the owner's budgets. Scope by user (or restrict budget routes to the primary user) before Budgets gets a UI. | Tier 3 |
| F11 | **Low** | **Alert latch is process memory, not data** (§9 says "the latch state is data") — every redeploy with a nonzero unparsed backlog re-pages. Persist per-alert latch state. | Tier 3 |
| F12 | **Low** | **No un-pair / correction path for transfer legs**, and `PATCH` kind-changes don't touch `transfer_group` — correcting one leg of a bad pair leaves a dangling group that violates the zero-sum invariant. Needed alongside F3's guards. | Tier 1 |
| F13 | **Low** | **Bill matching can bind to the wrong row**: pool is all transactions on the account, matched on `abs(amount)` — a same-magnitude deposit/refund can "pay" a bill, and two same-amount bills can claim one payment. Filter by sign/kind; enforce one bill per matched transaction. | Tier 3 |
| F14 | **Low** | **Scale/perf**: `_observation_history` loads the whole account table per evaluated row (O(N²) across a backfill, with regex normalization per comparison); `GET /transactions` has no pagination. Add SQL-side filtering + limit/offset before the backfill makes both hurt. | Tier 2 |
| F15 | **Low** | **Parser-replay is a column, not a capability** — `raw_payload` + `parse_version` are stored but no endpoint/script replays history through a fixed parser. Build the replay tool when the first parser fix needs it (or in Tier 3 with the bill parser). | Tier 3 |
| F16 | **Low** | **`_match_account` crashes the poll on last4 collision** (`scalar_one_or_none` raises on two accounts sharing last4), aborting the whole batch. Degrade to unparsed instead. | Tier 1 |
| F17 | **Low** | **Android `TokenStore` is plaintext** (no `EncryptedSharedPreferences`/Tink; Dragonfly encrypts its PAT, so the suite has the precedent). Decide deliberately for the finance app rather than inheriting. | Tier 4 |
| F18 | **Audit** | **Timezone/month-boundary semantics unaudited**: transaction dates come from parsers/CSV, "this month" comes from client-supplied year/month, `now()` is UTC — a late-evening swipe near month end may file under the wrong month vs. the owner's local time. One pass to pick and document the convention. | Tier 2 |

31. **Color grammar corrections (from the 2026-07-05 color critique).** The theme layer is
    right (four channels from proven Pulse hues, correct dark/light triples); the *usage*
    contradicts its own docstring:
    - **Red is over-loaded and dominates.** `overBudget` is documented as
      "over-budget/out-of-band deviations" but is used for every spend row (Transactions,
      review queue, Home spend tile) and every negative account balance — a credit card's
      normal statement balance renders as an alarm, and in a finance app ~80% of rows are
      spend, so the *rarest-should-be* color is the app's dominant one. Siblings reserve
      red for genuine errors. **Rule: ordinary spend and normal card balances are neutral
      (default/dim foreground); red appears only for over-budget, out-of-band, and missing
      states.**
    - **Bills colors the amount by status** (a paid bill's −$45 renders *green*, breaking
      the sign grammar the rest of the app teaches). Color the status word
      (Paid/Missing/Awaiting), never the amount.
    - **Teal means four unrelated things** — primary action, net/transfer, "awaiting
      payment", and AI-suggestion text — while the lead accent barely appears otherwise
      (no hero, no section identity). Plate's discipline is one hue = one domain. **Rule:
      teal = brand/primary + money-total (net, balances-in-good-standing); give AI drafts
      their own voice (violet is unclaimed inside Magpie and already reads
      "intelligence/system" in the suite); "awaiting" is neutral, not teal.**
    - **Channel density per card:** a review card currently carries amber border + red
      amount + amber note + teal AI text + teal button — four hues. Siblings hold cards to
      1–2 channels; the card's *context* channel (amber) plus at most one datum channel.
    - Keep: green = income/under-budget/reconciled (consistent today), amber =
      needs-review/warning, the local red channel built from `PulseRed`/`PulseRedDeep`
      rather than invented hues (that part was done right).
